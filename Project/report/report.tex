\section*{Introduction}

The New York City Taxi and Limousine Commission (TLC) publishes monthly trip records for four categories of for-hire vehicle service operating in New York City: Yellow taxi (traditional medallion cabs, primarily serving Manhattan), Green taxi (boro cabs introduced in 2013 to serve outer boroughs and upper Manhattan), FHV (for-hire vehicles such as black cars, limousines, and community cars dispatched through licensed bases), and FHVHV (high-volume for-hire vehicles -- specifically Uber and Lyft, which were required to report separately from February 2019 onward) \cite{tlc2024data}. Together these four datasets span 2009--2026 and contain approximately 3.7 billion trip records, representing one of the largest and most detailed publicly available urban mobility datasets in the world.

The scale and heterogeneity of this data -- varying schemas across file vintages, inconsistent column names and types, systematic data quality issues in some datasets, and a combined uncompressed size in the terabyte range -- make it a natural fit for big data processing techniques. This report covers a complete pipeline from raw ingestion through schema unification, data quality assessment, exploratory data analysis, storage format benchmarking, external data augmentation, market impact analysis, and geographic visualization.

\textbf{Platform.} All processing was performed on \textbf{Snowflake} \cite{snowflake2024}, which served as our unified compute and storage platform in place of an HPC cluster. Snowflake's elastic compute model, columnar storage, and native support for semi-structured data, geospatial operations, and Python stored procedures made it well-suited for this workload. SQL scripts handled all large-scale aggregations and transformations; Python notebooks running inside Snowflake's Snowpark environment were used for visualization, file format benchmarking, and external API data fetching. Data is organized in a three-layer medallion architecture: \texttt{BRONZE} (raw staged Parquet files), \texttt{SILVER} (unified and cleaned trip tables), and \texttt{GOLD} (aggregations, analysis outputs, and enriched tables).

All code is available in the accompanying repository, organized by task under \texttt{scripts/t0/} through \texttt{scripts/t9/}.

\section*{T0 \& T1 -- Data Ingestion and Schema Unification}

\subsection*{T0: Downloading Missing Files and Snowflake Setup}

The TLC dataset on the Arnes HPC cluster covered trips only up to January 2025. As of April 2026, data is available up to and including February 2026. We identified all missing monthly Parquet files for each of the four dataset types and downloaded them using a Python script (\path{scripts/t0/missing_data.py}) that issued parallel HTTP requests. Downloaded files were saved to a \texttt{data\_missing} directory on the HPC cluster.

On Snowflake, we created a dedicated warehouse (\path{BIGDATA_MZMB_WH}) and database (\path{BIGDATA_TAXI_MZMB}) with three schemas: \texttt{BRONZE}, \texttt{SILVER}, and \texttt{GOLD}. Four internal stages were created under \texttt{BRONZE} -- \path{YELLOW_RAW_STAGE}, \path{GREEN_RAW_STAGE}, \path{FHV_RAW_STAGE}, and \path{FHVHV_RAW_STAGE} -- and all Parquet files from both the original HPC directory and the newly downloaded \texttt{data\_missing} directory were uploaded using the Snowpark Python connector (\path{scripts/t0/snowflake.py}).

To replicate the map-reduce schema comparison task, we queried Snowflake's \path{INFORMATION_SCHEMA} to extract the column names and data types for each staged file using the \texttt{SOURCE\_FILE} metadata column as the partition key. The map step computed the schema per file; the reduce (finalizer) step computed a union of all distinct schemas across files. This produced four schema summary tables -- \path{BRONZE.T0_YELLOW_SCHEMA}, \path{BRONZE.T0_GREEN_SCHEMA}, \path{BRONZE.T0_FHV_SCHEMA}, \path{BRONZE.T0_FHVHV_SCHEMA} -- and a combined \path{BRONZE.T0_SCHEMA_ALL} view. The comparison revealed significant variation across file vintages: column names differed (\texttt{airport\_fee} vs.\ \texttt{Airport\_fee}), types differed (vendor ID stored as integer in some files and float in others), and entirely new columns appeared in later years (e.g.\ \texttt{cbd\_congestion\_fee} from 2024 onward).

\subsection*{T1: Silver Layer -- Schema Unification}

With the schema differences catalogued, we created four unified tables in the \texttt{SILVER} schema using the SQL in \path{scripts/t1/t1.sql}. Each table is clustered by \texttt{(PICKUP\_YEAR, PICKUP\_MONTH)} to enable efficient partition pruning in downstream queries.

Schema differences were resolved as follows. Type inconsistencies were handled using \texttt{TRY\_TO\_DOUBLE} and \texttt{TRY\_TO\_NUMBER}, which safely coerce values and return \texttt{NULL} on failure rather than raising an error. Column name aliases (where the same attribute appeared under different names in different file vintages) were resolved using \texttt{COALESCE}, returning the first non-null value across all candidate columns. A \texttt{PICKUP\_YEAR} and \texttt{PICKUP\_MONTH} attribute were derived from the pickup datetime using \texttt{YEAR()} and \texttt{MONTH()} and stored explicitly to support the cluster key. A \texttt{SOURCE\_FILE} column preserved the original filename for traceability.

The resulting unified tables are: \path{SILVER.YELLOW_TRIPS} (23 columns, ~1.3B rows), \path{SILVER.GREEN_TRIPS} (24 columns, ~84M rows), \path{SILVER.FHV_TRIPS} (10 columns, ~797M rows), and \path{SILVER.FHVHV_TRIPS} (28 columns, ~1.52B rows). At this stage no filtering was applied -- all records including those with quality issues are present, and data quality is addressed in T2.

\section*{T2 -- Data Quality Analysis}

\subsection*{Methodology}

Quality checks were implemented in a two-step map-reduce pattern in \path{scripts/t2/t2.sql}. The map step (\path{GOLD.*_QUALITY_FILE_YEAR}) computed issue counts per \texttt{(SOURCE\_FILE, PICKUP\_YEAR)} combination, distributing the computation across Snowflake's columnar micro-partitions. The reduce step (\path{GOLD.*_QUALITY_YEAR}) aggregated these file-level results by year to produce the final summary. A third long-format table (\path{GOLD.*_QUALITY_CHART}) pivoted the results into \texttt{(PICKUP\_YEAR, ISSUE\_TYPE, ISSUE\_COUNT, TOTAL\_ROWS)} rows suitable for visualization.

The following checks were applied to all datasets where the relevant columns existed:

\begin{itemize}[noitemsep,topsep=2pt]
  \item \textbf{Unexpected pickup year}: year outside the dataset's expected range (Yellow/Green: 2012--2026; FHV: 2015--2026; FHVHV: 2019--2026).
  \item \textbf{Pickup outside source month}: pickup timestamp more than 1 day outside the calendar month of the source file (allowing for delayed midnight entries at month boundaries).
  \item \textbf{Pickup equals dropoff}: identical pickup and dropoff timestamps.
  \item \textbf{Dropoff before pickup}: dropoff timestamp earlier than pickup timestamp.
  \item \textbf{Zero trip distance}: trip distance or trip miles equal to zero.
  \item \textbf{Non-positive passenger count}: passenger count $\leq 0$ (only for Yellow and Green, which have this column; null values were not flagged since many later records omit this field).
  \item \textbf{Negative fare}: fare amount or base passenger fare below zero.
  \item \textbf{Negative total amount}: total amount below zero.
\end{itemize}

FHV-specific checks additionally included missing pickup location, missing dropoff location, and missing dispatching base number. For FHVHV, non-positive trip miles, non-positive trip time, and negative driver pay were also checked.

\subsection*{Results -- FHV and FHVHV}

FHV presents the most severe data quality issues. In 2015 and 2016, virtually 100\% of all records have a dropoff timestamp that precedes the pickup -- over 63M affected rows in 2015 alone and 132M in 2016. This is a systematic issue with early FHV reporting, where dispatching bases submitted incomplete or zeroed-out dropoff timestamps. Missing location data is also pervasive: in 2015, 27\% of pickups and 98\% of dropoffs lack a location ID. These rates improve substantially from 2018 onwards as reporting requirements tightened, but even in 2023--2025 over 75\% of FHV pickups lack a location ID.

FHVHV is by far the cleanest dataset. Issue rates are below 0.1\% for most checks across all years. The dropoff-before-pickup rate is consistently around 8,000--12,000 rows per year out of 200--240M total -- less than 0.005\%. Negative base passenger fare drops from 0.25\% in 2019 (the dataset's first year) to near-zero by 2023--2025, suggesting improving data collection practices.

\subsection*{Clean Views}

Based on the quality analysis, four clean views were created in the \texttt{SILVER} layer. Yellow and Green apply strict filters: valid year range, pickup within source month ($\pm$1 day), dropoff after pickup, trip distance $>$ 0, non-negative fare and total, and passenger count either null or $>$ 0. FHVHV applies similarly strict filters adapted to its columns (trip miles $>$ 0, trip time $>$ 0, non-negative fare and driver pay, non-null locations). FHV intentionally applies only a light filter (valid year range and non-empty dispatching base) to avoid discarding the large fraction of early records with known systematic issues -- downstream analyses using FHV data simply work around the known problems rather than removing the data entirely.

\subsection*{Spatial Analysis}

Top pickup locations were aggregated in \path{GOLD.T3_TOP_PICKUP_LOCATIONS} and \path{GOLD.T3_TOP_DROPOFF_LOCATIONS}; the full location distribution is in \path{GOLD.T3_PICKUP_LOCATION_DIST}. Yellow taxi pickups are dominated by Manhattan zones (top: zone 237 Midtown, 161 Lincoln Square, 236 Murray Hill) with JFK and LaGuardia airport zones also appearing in the top 15. Green taxi is almost exclusively in the outer boroughs: top zones are 74 (East New York, Brooklyn), 75 (Flatbush, Brooklyn), and 41 (Crown Heights North, Brooklyn) -- reflecting its designed role as a boro cab. FHV and FHVHV show much wider geographic coverage, with zone 264 (representing trips with no location recorded, predominant in early FHV data) as an outlier at the top of the FHV list.

Origin-destination (OD) pair analysis (\path{GOLD.T3_TOP_OD_PAIRS}) reveals that self-loop pairs (same zone pickup and dropoff) are surprisingly common. The top Yellow OD pair is 264$\to$264 ($\sim$16M trips -- missing location data). Excluding this artefact, the most common genuine Yellow OD pairs are in the Midtown--Upper East Side corridor. Green's top pairs are local outer-borough trips, consistent with its service area.

Average fare and total amount have risen steadily across all three datasets with financial data, with a particularly sharp increase from 2022 onwards -- likely driven by post-pandemic demand recovery and fare structure adjustments. Average trip distance has remained relatively stable ($\sim$2.5--3.5 miles for Yellow and Green), while FHVHV averages slightly longer trips. Average trip duration increased noticeably around 2022--2023, which may reflect increased traffic congestion following the return to pre-pandemic activity levels.

\section*{T4 -- Storage Format Comparison}

\subsection*{Experimental Setup}

We selected the Green taxi 2024 year-partition as the benchmark target: \path{SILVER.GREEN_TRIPS_CLEAN WHERE PICKUP_YEAR = 2024}, yielding 617,885 rows across 24 columns. This partition is stored as approximately 16 MB in Parquet (snappy compressed), matching the task specification of a ``moderately sized'' partition.

Four formats were benchmarked: Parquet (snappy compression, via PyArrow \cite{arrow2024}), plain CSV (pandas), gzip-compressed CSV (pandas), and HDF5 with gzip compression (h5py). Snowflake natively supports Parquet, CSV, and gzip CSV via \texttt{COPY INTO} stage exports but does not support HDF5 or DuckDB, so those were benchmarked in Python only.

\subsection*{Measurement Methodology}

\textit{Python benchmarks} (\path{scripts/t4/t4_benchmark.ipynb}): The dataset was first loaded from Snowflake into a pandas DataFrame. Write time was measured as a single run. Read time was measured as the median of three consecutive full DataFrame reads from disk (to reduce the effect of OS filesystem caching on early runs). HDF5 was implemented using h5py directly: numeric columns were stored with gzip compression, string/datetime columns were serialised to fixed-length byte strings or int64 representations.

\textit{Snowflake benchmarks} (\path{scripts/t4/t4.sql}): Write time was measured as the duration of a \texttt{COPY INTO @GOLD.T4\_FORMAT\_STAGE/<format>/} query, retrieved from \path{INFORMATION_SCHEMA.QUERY_HISTORY_BY_SESSION}. Read time was measured as the duration of a \texttt{COUNT(*) FROM @stage/<format>/} query, which forces the engine to decompress and parse every byte of every file in the stage to count rows -- ensuring a consistent, full-scan comparison across formats. \texttt{SELECT *} was not used for staged CSV files because CSV embeds no schema metadata; Snowflake cannot infer column names from a CSV file alone and would require explicit positional column references (\texttt{\$1, \$2, \ldots}). Parquet embeds its schema and would support \texttt{SELECT *}, but \texttt{COUNT(*)} was used consistently across all formats.

\begin{figure}[h]
\centering

\caption{Storage format benchmark: file size (left), read speed (centre), and write speed (right) for all format/engine combinations.}
\label{fig:formats}
\end{figure}

\textbf{Parquet (snappy)} is the clear winner for analytical workloads: near-smallest file size (15.1 MB Python, 14.4 MB Snowflake), by far the fastest read in Python (0.11 s -- 11$\times$ faster than plain CSV), and fast write (0.82 s). Its columnar layout allows readers to skip entire columns not needed for a query, and snappy compression provides a good balance of compression ratio and CPU cost.

\textbf{CSV (gzip)} achieves the smallest file size (13.2 MB Python, 14.0 MB Snowflake), but at a significant cost: gzip compression is single-threaded and CPU-intensive, making write time 18.5 seconds in Python -- over 22$\times$ slower than Parquet. Read time is also slower than Parquet (1.40 s) because the decompression step must complete before any data can be processed. This format is appropriate for archival storage where write speed and read speed are secondary to file size.

\textbf{Plain CSV} is strictly dominated: it is the largest format (90--95 MB, 6$\times$ larger than Parquet) with no speed advantage over gzip CSV and significantly slower reads than Parquet. Its only practical advantage is universal compatibility with tools that cannot handle binary or compressed formats.

\textbf{HDF5} (Python-only) occupies a middle ground: 38.9 MB on disk and 0.30 s read time -- slower than Parquet to read but faster than CSV. Write time of 0.95 s is competitive. However, HDF5 is unavailable in Snowflake and requires more complex read code than Parquet or CSV, limiting its practical applicability in a Snowflake-centric pipeline.

\textbf{Snowflake vs.\ Python read times}: Snowflake's \texttt{COUNT(*)} read times (0.42--0.60 s) are slower than Python for the same formats. This is expected: the Snowflake measurement includes query planning, metadata lookup, and distributed execution overhead, none of which are present in a local Python \texttt{pd.read\_parquet()} call. Importantly, the relative ordering is preserved -- Parquet is still fastest for Snowflake reads, and CSV plain and CSV gzip are slower. Snowflake write times (2.1--3.2 s range) are slower than Python write times for Parquet and HDF5 because Snowflake must also encrypt and upload the data to its internal object storage.

\section*{T5 -- Data Augmentation}

\subsection*{Overview}

Each of the four cleaned trip tables was enriched with four categories of external contextual data: weather conditions at the time of pickup, the density of schools in the pickup zone, the density of tourist and cultural attractions in the pickup zone, and the number of permitted public events active during the pickup hour in the pickup borough. The final enriched tables are stored as \path{GOLD.T5_YELLOW}, \path{GOLD.T5_GREEN}, \path{GOLD.T5_FHV}, and \path{GOLD.T5_FHVHV}.

All external data was fetched from open APIs directly within Snowflake using Python stored procedures defined in \path{scripts/t5/t5_fetch_data.sql}, leveraging the \path{ALLOW_ALL_EAI} external network access integration. This allowed all data acquisition to occur within the Snowflake compute environment without requiring local execution or external ETL tools.

\subsection*{Taxi Zone Reference Tables}

Two zone reference tables underpin all spatial joins:

\path{EXTERNAL_DATA.TAXI_ZONE_LOOKUP}: The standard TLC taxi zone lookup CSV (downloaded directly from the TLC website) uploaded to Snowflake as-is. Provides \texttt{LOCATION\_ID}, \texttt{BOROUGH}, \texttt{ZONE}, and \texttt{SERVICE\_ZONE} for all 263 NYC taxi zones plus zone 264 (unknown/outside NYC).

\path{EXTERNAL_DATA.TAXI_ZONES_GEOM}: A geometry table for spatial point-in-polygon joins. The TLC taxi zone shapefile was downloaded and processed locally using \path{scripts/t5/convert_taxi.ipynb}: the shapefile was reprojected from its native EPSG:2263 (NY State Plane) to EPSG:4326 (WGS84) using GeoPandas \cite{geopandas2020}, each polygon's geometry was serialised to Well-Known Text (WKT) format, and the result was exported as \texttt{taxi\_zones\_wkt.csv}. This CSV was uploaded to Snowflake and the \texttt{WKT} column parsed into a native \texttt{GEOGRAPHY} type using \texttt{TO\_GEOGRAPHY()}, enabling efficient spatial operations via Snowflake's geospatial functions.

\subsection*{Weather Data}

Daily weather data for NYC (latitude 40.7831, longitude $-$73.9712) was fetched from the Open-Meteo historical weather archive API \cite{openmeteo2024}. Three variables were retrieved: daily maximum temperature (\texttt{temperature\_2m\_max}), daily minimum temperature (\texttt{temperature\_2m\_min}), and daily precipitation sum (\texttt{precipitation\_sum}), all in SI units. Data was fetched in five 3-year chunks (2012--2025) to stay within API request size limits. The stored procedure \texttt{EXTERNAL\_DATA.FETCH\_WEATHER()} handles chunked fetching, DataFrame construction, and table creation in a single call. Weather data is linked to trips by pickup date: \texttt{TO\_DATE(PICKUP\_DATETIME) = TO\_DATE(DATE)}.

\subsection*{Schools}

Primary and high school locations were fetched from the NYC Open Data portal (dataset identifier \texttt{n3p6-zve2}, ``2019-20 School Locations'') using the Socrata API. The stored procedure \texttt{EXTERNAL\_DATA.FETCH\_SCHOOLS()} retrieves up to 50,000 records, extracts latitude and longitude from the nested \texttt{location\_1} GeoJSON field, filters to coordinates within the NYC bounding box ($40.4$--$41.0$°N, $-74.3$--$-73.7$°W), and saves the result as \path{EXTERNAL_DATA.T5_SCHOOLS_RAW}.

Spatial joining to taxi zones was performed in \path{scripts/t5/t5.sql} using \texttt{ST\_CONTAINS(zone\_geom, ST\_POINT(longitude, latitude))}, producing \path{EXTERNAL_DATA.T5_SCHOOLS_BY_ZONE} with a school count per zone. This zone-level aggregate is then joined to trip tables by \texttt{PU\_LOCATION\_ID}.

\subsection*{Attractions}

Museums and cultural institutions were fetched from NYC Open Data (dataset \texttt{fn6f-htvy}, ``Museums'') following the same pattern. Coordinates were extracted from a \texttt{the\_geom} GeoJSON point field. After spatial join with taxi zone polygons, \path{EXTERNAL_DATA.T5_ATTRACTIONS_BY_ZONE} provides an attraction count per zone.

\subsection*{Events}

Permitted public events were fetched from NYC Open Data (dataset \texttt{bkfu-528j}, ``NYC Permitted Event Information'') for 2020--2024. The stored procedure \texttt{EXTERNAL\_DATA.FETCH\_EVENTS()} fetches data quarter by quarter (16 quarters total) to stay within the 10,000-row response limit per API request. Event borough and start/end datetimes are extracted; rows with missing start datetimes or boroughs are dropped.

Processing in \path{scripts/t5/t5.sql} creates \path{EXTERNAL_DATA.T5_EVENTS_HOURLY} by expanding each event across its active hours using \texttt{ARRAY\_GENERATE\_RANGE} and \texttt{LATERAL FLATTEN}: an event running from 14:00 to 17:00 generates entries for hours 14, 15, 16, and 17. When no end datetime is recorded, a default 3-hour duration is assumed. The result is aggregated to an active event count per \texttt{(HOUR, BOROUGH)} combination. Events are linked to trips by \texttt{DATE\_TRUNC('hour', PICKUP\_DATETIME) = ACTIVE\_HOUR} and \texttt{BOROUGH}.

\subsection*{Final Enriched Tables}

The gold tables are created by \path{scripts/t5/t5_gold.sql}, which joins each clean trip view against all four external data sources. Because the joins involve very large base tables (up to 1.52B rows), the script temporarily scales the Snowflake warehouse to \texttt{LARGE} before running and returns it to \texttt{XSMALL} afterwards. Each row in the output tables contains all original trip columns plus: \texttt{BOROUGH}, \texttt{ZONE}, \texttt{SERVICE\_ZONE} (from zone lookup), \texttt{TEMP\_MAX\_C}, \texttt{TEMP\_MIN\_C}, \texttt{PRECIPITATION\_MM} (from weather), \texttt{SCHOOL\_COUNT} (schools in pickup zone), \texttt{ATTRACTION\_COUNT} (attractions in pickup zone), and \texttt{ACTIVE\_EVENTS} (events active during pickup hour in pickup borough).

\section*{T8 -- Impact of FHVHV on Traditional Services}

\subsection*{Regulatory Context and Data Interpretation}

Before February 2019, Uber and Lyft operated in NYC as ordinary FHV dispatching bases, reporting their trips under the FHV dataset. When the NYC TLC's FHVHV (high-volume for-hire vehicle) regulation took effect in February 2019, operators dispatching more than 10,000 trips per day were required to obtain a new HVFHS license and report under the new FHVHV dataset. This caused an immediate and dramatic structural break: FHV monthly counts dropped from $\sim$24M in January 2019 to $\sim$1--2M in subsequent months, while FHVHV counts simultaneously emerged at $\sim$18--20M/month. Any longitudinal analysis must account for this break rather than treating FHV and FHVHV as independent time series.

Smaller rideshare operators also appeared briefly in FHVHV data: Juno (\texttt{HV0002}) and Via (\texttt{HV0004}) are visible with a small market share ($<$2\% each) around 2019 before disappearing -- Juno shut down operations in May 2019 after being acquired by Lyft, and Via exited the NYC consumer ride-hailing market. The analysis tracks six operator groups: Yellow taxi, Green taxi, Uber (\texttt{HV0003}), Lyft (\texttt{HV0005}), FHV (other, combining all non-Uber/Lyft FHV/FHVHV), Via, and Juno.

\subsection*{Absolute Trip Volumes}

Yellow taxi peaked at $\sim$170M trips/year in 2012--2013, when it held effectively 100\% of the tracked for-hire vehicle market (Green taxi was introduced only in late 2013). The onset of rideshare competition from 2014 began eroding Yellow's absolute volume: by 2018 it had fallen to $\sim$82M trips/year, a decline of $\sim$52\% from peak. The FHV category (which at this time included Uber and Lyft) grew rapidly over the same period, peaking at $\sim$260M trips/year in 2018.

After the FHVHV regulatory transition in 2019, Uber settled at $\sim$160--180M trips/year and Lyft at $\sim$50--65M. Yellow continued declining to $\sim$40M by 2023--2024 -- a 76\% reduction from its 2012 peak in absolute terms. Green taxi never exceeded $\sim$20M trips/year and has now fallen to near zero, driven out of its outer-borough niche by FHVHV services that offer lower prices and greater convenience.

\section*{T9 -- Geographic Visualization}

\subsection*{Rendering Methodology}

All maps were produced in \path{scripts/t9/t9_map.ipynb} using matplotlib. Zone polygon geometries were fetched from \path{EXTERNAL_DATA.TAXI_ZONES_GEOM} as GeoJSON strings via \texttt{ST\_ASGEOJSON(GEOM)}, then parsed in Python to extract coordinate lists for both \texttt{Polygon} and \texttt{MultiPolygon} feature types. Each zone was rendered as a matplotlib \texttt{Polygon} patch with a fill colour determined by its data value on a normalised colormap scale. Zones with no data for a given dataset were rendered in neutral grey. The map extent was fixed to the NYC bounding box ($-$74.27 to $-$73.70°W, 40.49 to 40.92°N) with equal aspect ratio.

\subsection*{T2 Data Quality Issues per Zone}

To produce a geographic view of data quality (\path{GOLD.T9_QUALITY_BY_ZONE}), the T2 quality checks were re-applied per pickup zone against the \textit{raw} (uncleaned) SILVER tables -- not the clean views -- so the map reflects the original data quality distribution before any filtering. Four issue types were aggregated per zone per dataset: zero trip distance (or trip miles for FHVHV), pickup equals dropoff, dropoff before pickup, and negative fare. FHV lacks distance and fare columns, so those issue counts are hardcoded to 0 for that dataset. The counts were combined into a single \texttt{TOTAL\_ISSUE\_PCT} score (total issue events as a percentage of total trips for that zone and dataset).

The colour scale (YlOrRd) was clamped to the 95th percentile of \texttt{TOTAL\_ISSUE\_PCT} across all zones to prevent a small number of extreme outlier zones from compressing the palette and obscuring variation in the bulk of the data.

\begin{figure}[h]
\centering
\includegraphics[width=\columnwidth]{t9/issues_map.png}
\caption{T2 data quality issue rate per taxi zone (YlOrRd, clamped to 95th percentile). Grey zones have no trip data for that dataset.}
\label{fig:quality_map}
\end{figure}

The geographic distribution of quality issues is consistent with the T2 findings. FHVHV is uniformly light (low issues) across the entire city. Yellow shows isolated hotspots in a small number of Manhattan zones and around the airports, with most of the city in the low-issue range. Green has scattered moderate-issue zones in specific outer-borough areas. FHV shows the most widespread elevated rates (orange/red) across large portions of the city -- particularly in zones that historically had many trips with missing location data and systematic dropoff timestamp problems.

A direct geographic mapping of the T4 storage format benchmark is not appropriate -- read and write speed are properties of the machine and file format, not of geographic location. \section*{Conclusion}

We designed and implemented a complete big data pipeline for 3.7 billion NYC TLC taxi trip records, using Snowflake as a unified compute and storage platform organized in a Bronze/Silver/Gold medallion architecture. The pipeline covers raw ingestion from heterogeneous Parquet files, schema unification across 14+ years of evolving data formats, systematic data quality assessment, exploratory analysis at scale, storage format benchmarking, multi-source external data augmentation, market impact analysis, and geographic visualization.

Several key findings emerge from this work. First, the February 2019 FHVHV regulatory transition was the single largest structural event in the dataset -- more impactful in terms of market reorganization than even COVID-19. It immediately transferred market leadership from FHV-reported rideshare to Uber and Lyft under a new regulatory category, collapsing FHV reporting volumes and crystallizing a new market structure that persists today. Second, COVID-19 caused a temporary but severe demand collapse in 2020 ($>$90\% for traditional taxis, $\sim$80\% for FHVHV), from which Yellow and Green taxi have recovered only partially -- settling at roughly one-quarter of their pre-pandemic volumes. Third, despite serving very different geographic areas and customer bases, all four service types share essentially identical temporal demand patterns (hourly and weekly), suggesting that the underlying urban mobility rhythm is uniform across service modes. Fourth, spatial analysis reveals sharp geographic complementarity between Yellow and Green taxis (spatial cosine similarity 0.08), contrasting with the broad geographic overlap between FHV and FHVHV (0.85). Fifth, Parquet with snappy compression is the optimal storage format for this analytical workload: it provides the fastest read times (11$\times$ faster than CSV in Python), competitive file sizes, and is natively supported across both Python and Snowflake environments.

The enriched GOLD tables produced by T5 -- joining trip records with weather, school density, attraction density, and event activity -- provide a foundation for downstream demand modelling and causal analysis of trip volume drivers. Future work could extend this pipeline to include streaming analysis (Kafka-based rolling statistics), machine learning for demand forecasting or anomaly detection, and distributed GPU computing for large-scale model training.
