\section*{Introduction}

Our goal in this project was to build a complete big data pipeline and analysis workflow using the New York City Taxi and Limousine Commission (TLC) trip record data. The TLC publishes monthly trip records for several types of taxi and for-hire vehicle services. We used the four datasets that were relevant for our analysis: Yellow taxi (medallion cabs, primarily concentrated in Manhattan), Green taxi (street-hail livery vehicles serving mainly northern Manhattan and the outer boroughs), FHV (for-hire vehicles dispatched through licensed bases), and FHVHV (high-volume FHV services such as Uber and Lyft, reported separately from 2019 onwards) \cite{tlc2024data}.

Most of our processing was done in \textbf{Snowflake} \cite{snowflake2024}, which served as our main big data platform. The Snowflake setup uses the warehouse:

\begin{center}
\texttt{BIGDATA\_MZMB\_WH}
\end{center}

and the database:

\begin{center}
\texttt{BIGDATA\_TAXI\_MZMB}
\end{center}

This is where all our main results are available, separated into the \texttt{BRONZE}, \texttt{SILVER}, \texttt{GOLD}, and \texttt{EXTERNAL\_DATA} schemas. The data pipeline follows a simple medallion structure: the \texttt{BRONZE} layer contains raw staged Parquet files and schema inspection outputs, the \texttt{SILVER} layer contains unified and cleaned trip tables, the \texttt{GOLD} layer contains aggregated results, enriched tables, and visualization inputs, and external datasets are stored in the \texttt{EXTERNAL\_DATA} schema. You can log in to Snowflake to see the results at \textbf{\path{https://sfedu02-fhb32629.snowflakecomputing.com}}, and then go to the mentioned database (Catalog $\to$ Database Explorer).

The full project is also available on GitHub at: \textbf{\path{https://github.com/7matic/bd_project/tree/main/Project}}.

The repository contains all scripts, notebooks, SQL files, figures, and other report materials used for the project. Code is organized by task under \path{scripts/}, while figures used in the README and report are stored under \path{images/}.

In the following sections, we go through all the tasks and describe them individually, following the main CRISP-DM approach: first understanding, preparing, and exploring the data (earlier stages), then using that information to do proper modeling and evaluation.

\section*{T0 \& T1: Data ingestion and schema unification}

\subsection*{T0: Downloading, staging, and schema inspection}

We initially used the shared TLC dataset on the Arnes HPC cluster at \path{/d/hpc/projects/FRI/bigdata/data/Taxi}. Since we already had the Parquet files up to January 2025, it was easier to obtain the remaining newer files. We downloaded the missing data up to February 2026 for all four datasets: Yellow taxi, Green taxi, FHV, and FHVHV. Missing files were downloaded using a Dask Bag workflow in \path{scripts/t0/missing_data.py} and saved on the same HPC cluster. We applied the map-reduce approach: the map step processed files independently, and the final step summarized how many files were downloaded, present, or missing.

After obtaining all the necessary files, we moved to Snowflake and created our main warehouse and database: \texttt{BIGDATA\_MZMB\_WH}, \texttt{BIGDATA\_TAXI\_MZMB}. Within the \texttt{BRONZE} schema, we created one internal stage per dataset: \texttt{YELLOW\_RAW\_STAGE}, \texttt{GREEN\_RAW\_STAGE}, \texttt{FHV\_RAW\_STAGE}, and \texttt{FHVHV\_RAW\_STAGE}. We used the upload script \path{scripts/t0/snowflake.py} to collect files from both the original shared directory and the missing-data directory where we downloaded new files, and then uploaded them to the correct Snowflake stage using the PUT command.

For schema inspection, we saved results in four tables, one per dataset, and also combined them into \path{BRONZE.T0_SCHEMA_ALL} to provide an overview of all obtained column names, data types, null information, and source files. Raw stages and the created inspection tables can be seen in Figure \ref{fig:t0}.

\section*{T2: Data quality analysis}

\subsection*{Methodology}

After creating the unified \texttt{SILVER} tables in T1, we inspected the data quality issues common in the TLC files. The main script for this task is \path{scripts/t2/t2.sql}. For each dataset, we created three result tables in the \texttt{GOLD} schema:
\begin{itemize}[noitemsep,topsep=2pt]
\item \texttt{QUALITY\_FILE\_YEAR}: file-level quality statistics grouped by source file and pickup year,
\item \texttt{QUALITY\_YEAR}: yearly aggregation of the file-level results,
\item \texttt{QUALITY\_CHART}: table used for plotting issue percentages.
\end{itemize}

The per-file table represents the map step, and the yearly table represents the reduce step. The charts were created directly from the \texttt{QUALITY\_CHART} tables in Snowflake, which is a useful feature that allows for quick visualizations directly on the platform. We used percentages in the visualizations instead of raw counts because the datasets vary greatly in size.

The exact checks regarding data quality depended on the dataset, because not all datasets have the same columns. The main checks were:
\begin{itemize}[noitemsep,topsep=2pt]
\item pickup year outside the expected range for the dataset,
\item pickup timestamp outside the source file month, with a one-day tolerance,
\item pickup timestamp equal to dropoff timestamp,
\item dropoff timestamp earlier than pickup timestamp,
\item zero or non-positive distance,
\item non-positive passenger count for Yellow and Green,
\item negative fare or total amount,
\item missing pickup/dropoff locations and missing base information for FHV/FHVHV.
\end{itemize}

\subsection*{Comparison metrics}

To compare how similarly the four datasets behave, we use cosine similarity for distribution-shaped data. Cosine similarity measures the angle between two vectors rather than their magnitude, so it captures whether two distributions have the same shape regardless of how many total trips each dataset has. For monthly volume, where we care about correlated trends over time rather than shape, we use Pearson correlation instead.

\section*{T4: Storage format comparison}

\subsection*{Experimental setup}

We benchmarked five storage formats using the Green 2024 partition (617,885 rows, 24 columns) from \path{SILVER.GREEN_TRIPS_CLEAN}: Parquet with snappy compression (PyArrow \cite{arrow2024}), plain CSV (pandas), gzip-compressed CSV (pandas), HDF5 with gzip compression (h5py), and DuckDB. Snowflake supports Parquet, CSV, and gzip CSV natively via \texttt{COPY INTO} stage exports. HDF5 and DuckDB were benchmarked locally only, as neither can be installed in Snowflake's Snowpark environment.

\subsection*{Measurement methodology}

For Python, write time was measured as a single run and read time as the median of three runs from disk. HDF5 used h5py with gzip compression for numeric columns and fixed-length byte strings for text and datetime fields.

For Snowflake, write time was the duration of the \texttt{COPY INTO} query and read time was the duration of a \texttt{COUNT(*)} scan on the staged files. A full \texttt{COUNT(*)} was chosen over \texttt{SELECT *} because staged CSV files carry no schema metadata. Snowflake cannot infer column names without explicit positional references, so \texttt{COUNT(*)} was used consistently across all formats to force a full parse.

\subsection*{Results}

Table~\ref{tab:formats} and Figure~\ref{fig:formats} show the benchmark results.

\begin{table}[h]

\midrule

\begin{figure*}[t]
\centering
\includegraphics[width=\textwidth]{t4/grapf_comparison.png}
\caption{Format benchmark: file size, read speed, and write speed for all format/engine combinations.}
\label{fig:formats}
\end{figure*}

Parquet (snappy) is the clear winner for analytical workloads: 0.11 s read time (11$\times$ faster than plain CSV), a competitive 15.1 MB file size, and a fast 0.82 s write. Its columnar layout lets readers skip columns they do not need, and snappy compression is cheap to decompress.

Gzip CSV achieves the smallest file size (13.2 MB) but at a steep cost: write takes 18.5 s, over 22$\times$ slower than Parquet, because gzip compression is single-threaded and CPU-intensive. This format makes sense only for long-term archival where read and write speed are secondary concerns.

Plain CSV is the weakest option overall, it is 6$\times$ larger than Parquet with no speed advantage. Its only practical benefit is broad tool compatibility.

HDF5 sits in the middle: 38.9 MB, 0.30 s read, and 0.95 s write. It is a reasonable choice for Python-only workflows, but it is unsupported in Snowflake and requires more complex code to read and write than Parquet or CSV.

DuckDB was benchmarked locally since it cannot be installed in Snowflake's Snowpark environment. It produced a 23.6 MB file, read in 0.62 s, and write in 1.76 s. This puts it between HDF5 and Parquet on read speed, and between Parquet and HDF5 on file size. DuckDB is a strong option for local analytical workflows where the full Snowflake stack is not available, but its embedded file format is not portable across different tools the way Parquet is.

Snowflake read times (0.42--0.60 s) are higher than the equivalent Python reads because they include query planning and distributed execution overhead. The relative ordering across formats is the same as in Python, with Parquet still the fastest. Snowflake write times are also higher than local Python writes because Snowflake encrypts and uploads data to its internal object store.

\section*{T5: Data augmentation}

Each cleaned trip table was enriched with four categories of external contextual data: weather at pickup time, school density in the pickup zone, attraction density in the pickup zone, and the number of active permitted events in the pickup borough. All data was fetched directly inside Snowflake using Python stored procedures with external network access (\path{scripts/t5/t5_fetch_data.sql}), so no local ETL was required. The final enriched tables are stored as \path{GOLD.T5_YELLOW}, \path{GOLD.T5_GREEN}, \path{GOLD.T5_FHV}, and \path{GOLD.T5_FHVHV}.

\subsection*{Taxi zone reference tables}

Two reference tables are used in all spatial joins. \path{EXTERNAL_DATA.TAXI_ZONE_LOOKUP} is the standard TLC zone lookup CSV, downloaded directly from the TLC website and loaded into Snowflake as-is. It maps each of the 263 location IDs to a borough, zone name, and service zone. \path{EXTERNAL_DATA.TAXI_ZONES_GEOM} holds the actual polygon geometry for each zone, used for point-in-polygon lookups. The TLC shapefile was reprojected from EPSG:2263 to EPSG:4326 using GeoPandas (\path{scripts/t5/convert_taxi.ipynb}), serialised to WKT, and loaded into Snowflake where the geometry column was created with \texttt{TO\_GEOGRAPHY()}.

\subsection*{External data sources}

Daily weather data for NYC was fetched from the Open-Meteo historical API \cite{openmeteo2024} in five 3-year chunks to stay within request size limits. Three variables were retrieved, daily maximum temperature, minimum temperature, and precipitation sum, and then joined to trips by pickup date.

School locations were fetched from the NYC Open Data portal (dataset \texttt{n3p6-zve2}, 2019--20 School Locations) via the Socrata API. Coordinates were extracted from the nested GeoJSON response and each school was spatially joined to its taxi zone using \texttt{ST\_CONTAINS()}. The result is a school count per zone stored in \path{EXTERNAL_DATA.T5_SCHOOLS_BY_ZONE}.

Museum and attraction locations were fetched from NYC Open Data (dataset \texttt{fn6f-htvy}) using the same approach. The result is an attraction count per zone stored in \path{EXTERNAL_DATA.T5_ATTRACTIONS_BY_ZONE}.

Permitted public event data was fetched from NYC Open Data (dataset \texttt{bkfu-528j}) for 2020--2024 in quarterly chunks of 16 requests to respect the 10,000-row API limit \cite{nycopen2024}. Each event was expanded across its active hours in \path{scripts/t5/t5.sql} using the functions \texttt{ARRAY\_GENERATE\_RANGE} and \texttt{LATERAL FLATTEN}. The events with no recorded end time were assumed to last 3 hours. The result is an active event count per hour and borough, joined to trips by truncated pickup hour and borough.

\subsection*{Final enriched tables}

The gold join script (\path{scripts/t5/t5_gold.sql}) temporarily scales the Snowflake warehouse to \texttt{LARGE} to handle base tables of up to 1.52B rows. \section*{T6: Streaming with Kafka}

We treated the historical trip information as a stream and chose the year 2021 to combine the two data sources: Yellow Taxi and the FHVHV dataset. Because using the streaming API and streaming into Snowflake adds additional costs, the producer was limited to a smaller sample, especially since we had to experiment with several approaches to make it work on Snowflake and did not want to increase costs by too much.

The streaming setup was implemented using Docker (configuration located in \path{scripts/t6/docker-compose.yaml}) and Kafka, which included two Kafka brokers and a Kafka Connect service. Kafka Connect is used with the Snowflake Kafka Connector to ingest Kafka messages directly into Snowflake through the Snowpipe Streaming workflow.

We worked with two Kafka topics, \path{t6-yellow-2021} and \path{t6-fhvhv-2021}, so we could replay the records and monitor them independently. In Snowflake, we first created \path{GOLD.T6_STREAM_SOURCE_2021}, a table that combines records from \path{SILVER.YELLOW_TRIPS_CLEAN} and \path{SILVER.FHVHV_TRIPS_CLEAN} into a common schema. We also added two extra columns: \path{dataset} and \texttt{stream\_id}, which is based on pickup time order. The producer (\path{scripts/t6/producer_t6_2021.py}) reads this table, ordered by \path{stream_id}, to send each row as a JSON message to the correct Kafka topic in proper order, allowing us to simulate the streaming process.

To transfer data from Kafka to Snowflake, the Kafka Connector first writes messages to \path{GOLD.T6_KAFKA_LANDING_2021}, storing the JSON message in \path{RECORD_CONTENT} and key metadata (topic, partition, offset) in \path{RECORD_METADATA}. This table contains raw JSON, which is then parsed into standard columns in the view \path{GOLD.T6_STREAM_EVENTS_2021}.

The next task was stream analysis, for which we aggregated events into hourly buckets and then computed rolling statistics over 24 hours, where each window included the current hour and the previous 23 hours. We calculated the count, mean, and standard deviation for the three key attributes: \path{trip_distance}, \path{trip_duration_sec}, and \path{fare_amount}. These calculations were performed separately for each borough and for the top 10 most active locations, which we selected by combining the number of pickups and dropoffs and choosing the most active ones. Figure \ref{fig:t6stats} shows a preview of the rolling 24-hour statistics created in Snowflake. 

\begin{figure}[h]
\centering
\includegraphics[width=1\columnwidth]{images/t6/rolling_borough_stats.png}
\includegraphics[width=1\columnwidth]{images/t6/rolling_top_locations_stats.png}
\caption{Rolling 24-hour statistics by borough (top) and for the top 10 locations (bottom).}
\label{fig:t6stats}
\end{figure}

The last part of this task was to apply stream clustering, for which we implemented a Kafka consumer (\path{scripts/t6/consumer_t6_birch_2021.py}) that subscribes to both Kafka topics and processes messages. We chose the BIRCH algorithm, which is suitable for this task because it supports incremental fitting with partial\_fit, allowing us to update the model without loading the entire stream in advance. BIRCH incrementally builds a compact clustering-feature tree, summarizing similar points into subclusters. First, we performed feature engineering to convert our features of interest into numeric values (dataset flag, cyclic hour and weekday encodings, pickup and dropoff location IDs, trip distance, trip duration, and fare amount).

During streaming, we trained the model with \texttt{n\_clusters=None}, which allows it to build the internal tree structure incrementally. We then summarize the obtained results into final clusters, which we set to 6 in our case, so the results are quickly interpretable. Results are saved as \path{CSV} files in \path{scripts/t6} and also written back to Snowflake as tables. Figure \ref{fig:t6birch} shows the final summarized results of the BIRCH stream clustering.

\begin{figure}[h]

From the figure, we can see that some clusters consist entirely of Yellow taxi trips and are oriented toward Manhattan or the airports, while others are dominated by FHVHV and are more dispersed. This indicates some overlap, but also distinct usage patterns.

\section*{T7: City-wide demand forecasting}

\subsection*{Methodology}

For the batch machine learning project, we chose the city-wide demand forecasting problem. Our goal was to predict hourly ride demand using information from all four datasets. This allows for better resource planning and helps us understand how demand changes depending on the type of service.

Using \path{scripts/t7/t7.sql}, we created our main feature table, \texttt{GOLD.T7\_DEMAND\_FEATURES}. Each entry represents one hourly demand observation for a service and a pickup zone: \texttt{dataset + pickup\_location\_id + pickup\_hour}. The target was \\ \texttt{TRIP\_COUNT}, but for training, we transformed it to \texttt{LOG\_TRIP\_COUNT = LN(1 + TRIP\_COUNT)} because demand is highly skewed, which means that many zone-hour combinations have low demand, while some have very high demand. For predictions, we scaled back to the original scale to have a fair evaluation.

Because we are working with temporal data, the data split for training and evaluation must be based on time, not random selection. For the training set, we used the common hourly range where all four datasets and the T5 weather data were available, starting in 2019, up to the beginning of 2024. For the validation set, we used data from 2024, and for the test set, data from 2025.

We also wanted to see how much the augmented feature set could influence and improve the final predictions. The baseline set included features such as service type, pickup zone, time features (weekend, rush-hour, night flags), cyclic encodings (hour, day, month cyclic features), lag features (1h, 24h, 168h), and rolling averages (24h, 168h). The augmented set included all of these, and also added external data from the T5 task, such as information about weather, number of schools, attractions, and active events.

For algorithm selection, we chose \texttt{XGBoost} as a native distributed third-party algorithm and \texttt{SGDRegressor.partial\_fit} to represent scikit-learn algorithm with partial\_fit. The third algorithm we included was \texttt{PyTorch DDP} (Distributed Data Parallel), implemented as a simple custom neural network in PyTorch. This was an alternative to the Dask-ML category, which we could not install or run in the Snowflake environment. Since PyTorch usage was documented under the distributed training on Snowflake, we included it for additional comparison. To perform the task in a distributed manner, XGBoost used Snowflake ML data connectors, PyTorch used sharded data connectors, and SGD read the data in batches.

We did our distributed experiments in a Snowflake Container Runtime using a Ray cluster, which is Snowflake's method for scaling and utilizing distributed capabilities. It runs as a background process that we can easily connect to and scale dynamically during development. In our testing, we scaled the cluster to different numbers of workers and measured training time, prediction time, error metrics, process memory, and GPU memory.

\section*{T8: Impact of FHVHV on traditional services}

\subsection*{Regulatory context}

Before February 2019, Uber and Lyft reported under the FHV dataset. The FHVHV regulation required operators dispatching more than 10,000 trips per day to obtain a new HVFHS licence and report separately. FHV monthly counts immediately dropped from around 24M to 1--2M, while FHVHV simultaneously emerged at 18--20M trips per month. Any analysis must treat this as a structural break, not a real volume change. Two smaller operators, Juno (\texttt{HV0002}) and Via (\texttt{HV0004}), briefly appeared in FHVHV data with under 2\% share before exiting NYC in 2019.

\subsection*{Market share evolution}

Yellow taxi peaked at around 170M trips per year in 2012--2013 with effectively 100\% market share. By 2018 it had fallen to around 82M ($-$52\%) and by 2023--2024 to around 40M, a 76\% drop from peak. Green never exceeded 20M trips per year and has since fallen to near zero. After the 2019 transition, Uber settled at 160--180M trips per year and Lyft at 50--65M.

Figure~\ref{fig:stacked} shows the stacked annual share. Yellow held close to 100\% in 2012, fell below 50\% by mid-2016, and was at around 22\% just before the FHVHV transition. Uber immediately captured around 50\% of all tracked trips in 2019 and has since stabilised at 55--60\%. Lyft holds a steady 20--22\%, Yellow around 13--15\%, and Green below 1\%.

\section*{Conclusion}

We built a complete big data pipeline for approximately 3.7 billion NYC TLC trip records, including ingestion, cleaning, exploration, augmentation, streaming, machine learning, and visualization. The main data finding is that NYC ride demand changed in two significant ways. The February 2019 FHVHV reporting change separated Uber, Lyft, and similar services from the FHV category, making app-based traffic visible as its own market. The second major change was COVID-19, which caused a significant loss of volume, especially for Yellow and Green taxis, which never fully recovered. We observed that demand still follows almost the same daily and weekly rhythms, but spatial patterns are very different, as also seen during BIRCH stream clustering. Yellow taxis are mainly concentrated in Manhattan and airport-related areas, Green taxis operate outside Manhattan's core, and FHV/FHVHV services have much broader citywide coverage. Technical results showed that Parquet with Snappy compression was the best storage format for our workload, and distributed XGBoost was the best forecasting model. We also found that the best XGBoost results came from training separate models for each dataset. External augmentation provided richer features, but did not improve every machine learning model.

\subsection*{Experience with Snowflake}

We think that using Snowflake was a good choice, and overall we had a very positive learning experience with it. We completed the regular homework assignments, which included almost the same tasks, entirely on the Arnes HPC cluster, and found that Snowflake was generally much more intuitive and interactive. On the HPC cluster, you work directly with large Parquet files, which usually requires preparing scripts, submitting jobs, waiting for results, and then separately checking the results. In Snowflake, you interact directly with tables; the data is always available, and you can easily inspect and perform many visual checks, which are almost immediate in the web interface.

It also works very well as a big data platform because you have elastic warehouses that you can scale up for expensive joins and scale down for regular queries. There are also many useful built-in functions (for example, for geospatial analysis in T5 and T9) and SQL features that help replace much of the custom Python logic. For the entire execution of the project, we used 32.3 compute credits, and our total Snowflake storage was 203.9 GB.

There are still some difficulties, and the main one is that Snowflake is a complex ecosystem with its own learning curve. You immediately notice that it uses its own version of SQL, which is close to standard SQL, but some operations must be done in a Snowflake-specific way, so you need to learn those patterns. This is helpful in the long run, but it takes time to discover everything and learn how to use features effectively so they actually benefit you. The documentation can sometimes be overwhelming and cause information overload, either because there are not enough examples or because there are too many different ways to solve a similar problem. The hardest part was the machine learning, as it required extensive documentation review to set up distributed training of ML models and properly use the Ray cluster, so the T7 and T10 tasks required the most experimentation and time to solve. Other limitations included that Snowflake does not directly support every format or workflow we wanted to benchmark (such as HDF5 and DuckDB), so we performed those tasks in Python. Streaming also introduces additional complexity and costs, so we worked with a limited streamed sample during development to avoid unnecessary expenses. 

Overall, we are very satisfied with our decision to work with Snowflake and appreciate the opportunity to learn more about this extremely powerful platform. It made it much easier to combine large-scale SQL processing, data organization, visualizations, external data integration, and machine learning within a single unified environment. While it does take more time to learn initially, and as tasks become more complex its disadvantages start to appear, for simpler tasks and most analytical work it is a highly beneficial platform that accelerates workflow. There are also many features we did not need for this project, so there is still much we do not know and could use for other types of projects.
